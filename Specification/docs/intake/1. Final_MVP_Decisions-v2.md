# Final MVP Decisions

## Doctor-Patient Linking
Use invite code linking immediately.  
Do not rely only on manual Supabase linking.  

### Flow:
1. Doctor logs in.
2. Doctor opens "Add patient."
3. Doctor creates invite code.
4. App generates short single-use code.
5. Doctor gives code to patient.
6. Patient logs in.
7. Patient opens "Link Doctor" screen.
8. Patient enters invite code.
9. App creates active doctor-patient access row.
10. Doctor can now see patient in Doctor Dashboard.

### Use table: `doctor_invite_codes`
**Fields:**
* `id`
* `doctor_id`
* `code`
* `status`
* `expires_at`
* `max_uses`
* `used_count`
* `created_at`
* `used_at`

**Code rules:**
* Single-use for MVP.
* Expires after 7 days.
* Can be revoked by doctor before use.
* Patient should later be able to revoke doctor access.

### Use table: `doctor_patient_access`
**Fields:**
* `id`
* `doctor_id`
* `patient_id`
* `status`
* `created_at`
* `revoked_at`

**Status values:**
* `active`
* `revoked`

*Doctor can only view patients with active access.*

---

## Account Roles
* One account has one role.
* **Roles:**
    * `patient`
    * `doctor`
* For MVP, one user cannot switch roles inside the same account.
* If the girlfriend needs both doctor and patient access, she should create two separate accounts:
    * one doctor account
    * one patient account
* Her patient account can be linked to her doctor account using invite code just like any other patient.

---

## Offline-Lite Mode
Use offline-lite mode.

**Offline mode should support:**
* viewing cached own history
* opening previously cached days
* creating text entries
* editing timestamps for pending entries
* saving entries locally
* showing "Pending sync"
* syncing pending entries when online returns

**Offline mode should not support:**
* photo upload
* fresh export with signed image URLs
* fetching new doctor/patient data
* complex multi-device conflict resolution

*Voice input offline is best-effort only.* If device/browser speech recognition does not work offline, fallback to manual typing.

**Photo behavior offline:**
* User can create text entry offline.
* User cannot upload photo offline.
* User can later open previous day/entry when online and attach photo.

---

## Photo Rules
Photo system must support 3-5 users on Supabase free tier.

**Rules:**
* Max image width: 1280px
* Format: JPEG
* JPEG quality: 0.8
* Thumbnail: yes
* Upload original full-size image: no
* Private bucket: yes
* Photo upload offline: no
* Photo can be added later to old entry: yes

### Photo upload flow:
1. User selects/takes photo.
2. App previews image.
3. App resizes/compresses image client-side.
4. App generates thumbnail.
5. App uploads compressed image and thumbnail to private Supabase Storage.
6. App stores paths and metadata in database.
7. App does not store base64 image data in Postgres.
8. App does not upload original image.

**Target file sizes:**
* Main photo target: 250-500 KB if possible.
* Thumbnail target: 20-60 KB if possible.

**Add storage protection:**
* Track estimated storage usage.
* Show warning if project/user is approaching storage budget.
* Do not allow accidental huge uploads.
* Bucket should reject files above a safe per-file limit.

---

## Three-Month Research Retention
Do not delete data during the 3-month research period.  
All patient entries, photos, and metadata should remain available for the full 3 months.  

**After each month:**
* Doctor can export that month's JSON.
* Doctor can optionally download/archive photos.
* Data should still remain in Supabase until the 3-month research is finished.

**After 3 months:**
* Export full 3-month dataset.
* Archive JSON and photos externally if needed.
* Then optionally delete old photos/data from Supabase to free storage.

*Do not build automatic deletion in MVP.* **Add future feature:** storage cleanup/archive tool after research period.

---

## Export Rules
Only doctor account has export features in MVP.  
Patient does not need export buttons in MVP.  

**Doctor can export:**
* selected day
* selected month

*Month export should not require the month to be finished.* *Example:* If today is June 6 and doctor exports June, export June 1-June 6 data.

**Export range logic:**
* Day export: selected date 00:00 to next day 00:00
* Month export: selected month start to either current date/time or month end, whichever comes first

### Export modes:
1. **All data**
2. **All data with images**
3. **Images only with labels**

### Mode details:
* **All Data**
    * Include: patient code, date range, all entries, timestamps, entry type, title, note text, custom fields, doctor notes if selected.
    * No image signed URLs required unless specifically included.
* **All Data With Images**
    * Include: everything from All Data, photo metadata, signed image URLs, thumbnail URLs if useful.
* **Images Only With Labels**
    * Include only: image signed URLs, image metadata, timestamp, entry label/title, entry type, label for what the image belongs to.
    * *Examples of labels:* Breakfast, Lunch, Dinner, Snack, Symptom photo, Burn, Rash, Other custom label.
    * *Purpose:* This mode is for AI analysis from images only, but still with minimal labels so AI knows what the image is attached to.
    * Do not include long descriptions or symptom notes in images-only mode.
    * Do not embed base64 image data in JSON.

## Input Fields / Form System
Use preexisting entry types and fields for MVP, but allow adding extra fields.  
The app should have predefined entry types: **Meal, Symptom, Medication, Exercise, Daily note, Custom**.  
Each entry type can have basic predefined fields.  

There should also be an **"Add field"** button.  
When adding a field, user selects what the field belongs to.  
*Example:* belongs to Meal, belongs to Symptom, belongs to Medication, belongs to Exercise, belongs to Custom.  

The app should reorganize/display fields based on selected category/type.  

**Implementation idea:**
* Predefined forms for common entry types.
* Extra fields stored in `custom_fields` JSONB.
* Dynamic form builder is not required in full MVP.
* Full doctor-configurable form builder can be added later.

---

## Voice Input
Use free voice input only.  
No paid AI transcription API.  
No OpenAI Whisper API.  
No raw audio storage.  

**Voice input should:**
* be reusable across all text fields
* support Serbian and English where available
* use device/browser speech recognition
* append transcript to existing text
* allow user to edit before saving
* fallback to typing if unavailable
* work on Android, Huawei where supported, and web where supported

**Voice languages:**
* Serbian: `sr-RS`
* English: `en-US`

---

## UI / UX Style
Use clean medical-style UI.  

**Requirements:**
* Simple, readable, large buttons, older-user friendly, mobile-first, web-compatible, not flashy, not game-like, Serbian/English UI, light mode, dark mode, Figma workflow.
* Screens should be designed first as simple wireframes in Figma, then implemented.
* Do not over-polish before core flows work.

---

## Language and Theme
Support two UI languages from the start: **Serbian and English**.  

**Add Settings options:**
* App language: Serbian / English
* Voice language: Serbian / English
* Theme: Light / Dark, optionally System later

*Use translation files.* Do not hardcode UI text in components.

---

## Consent Screen
Add simple consent/privacy screen.  

**Text should explain:**
* app is for private tracking/research testing
* app is not a diagnosis tool
* data may include food, symptoms, notes, and photos
* doctor account may review linked patient data
* data can be exported for research analysis
* user can request deletion of data

*User must accept consent before using app.*

---

## Testing Targets
* Test on: Android phone, Huawei Android phone, web app in browser.
* For web voice input, prioritize Chromium-based browsers first.
* Voice may not work equally in every browser.

---

## Definition of Done for V1
V1 is done when:
* Patient can login.
* Doctor can login.
* Account role is respected (one account has one role).
* Doctor can generate invite code.
* Patient can link doctor using invite code.
* Doctor can see linked patient (cannot see unlinked patients).
* Patient can create online text entry.
* Patient can create offline text entry.
* Pending offline entry syncs when online.
* Patient can edit timestamp.
* Patient can view own timeline/history.
* Patient can add photo online.
* Patient can add photo later to an old entry.
* Photo is compressed before upload.
* Thumbnail is generated.
* Original photo is not uploaded.
* Private Supabase bucket is used.
* Voice input works where device/browser supports it.
* Typing fallback works.
* Serbian/English UI switch works.
* Light/dark mode works.
* Doctor can export selected day JSON.
* Doctor can export current/partial month JSON.
* Export modes exist: all data, all data with images, images only with labels.
* RLS prevents unrelated data access.
* Android APK works, Huawei APK works, web app works.
