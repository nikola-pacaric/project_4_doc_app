# Supabase Contract: V1 Patient Research Tracking App

## Purpose

This contract defines the expected Supabase backend boundaries for V1. Exact SQL,
policy names, and function implementation details belong to implementation, but
the behavior below is required.

## Authentication Contract

- Public signup creates patient accounts only.
- Doctor accounts are manually created by an administrator in Supabase.
- Each authenticated account has exactly one profile row and exactly one role.
- Client apps must never be able to assign or change a profile role to doctor.
- Consent acceptance is required before patient or doctor workflows open.

## Table Contract

Required logical tables:

- `user_profiles`
- `patient_baseline_profiles`
- `patient_entries`
- `daily_form_details`
- `meal_details`
- `symptom_details`
- `stool_details`
- `medication_details`
- `exercise_details`
- `menstruation_events`
- `entry_photos`
- `doctor_invite_codes`
- `doctor_patient_access`
- `export_requests`
- `audit_events`

Implementation may combine detail tables when justified, but contracts and
exports must preserve the same logical fields.

## Row-Level Security Contract

RLS must be enabled on all exposed app tables.

Patient policies:

- Patients can read their own profile, baseline, entries, details, photos,
  settings, and active doctor links.
- Patients can insert/update/delete their own entries and details.
- Patients can delete their own photos and photo metadata.
- Patients cannot read or write any other patient's data.

Doctor policies:

- Doctors can read their own profile.
- Doctors can read active invite codes they created.
- Doctors can create and revoke their own unused invite codes.
- Doctors can read profiles and patient data only for active
  `doctor_patient_access` links.
- Doctors cannot update/delete patient-created entries, details, or photos.
- Doctors can create export requests only for active linked patients.

Unauthenticated policies:

- No app data is visible to unauthenticated users.

Admin/service policies:

- Admin/service operations may create doctor profiles, repair data, and perform
  privileged export/linking operations.
- Service-role keys must not be included in web or mobile clients.

## Storage Contract

Bucket:

- One private bucket for patient entry photos and thumbnails.

Path convention:

- `patients/{patient_id}/entries/{entry_id}/photos/{photo_id}.jpg`
- `patients/{patient_id}/entries/{entry_id}/thumbs/{photo_id}.jpg`

Rules:

- Only authenticated owners may upload to their own patient folder.
- Linked doctors may read linked patient images through authenticated access or
  time-limited signed references generated by a guarded server/RPC path.
- Original full-resolution files must not be stored.
- Storage object metadata must correspond to `entry_photos` rows.
- Per-file size limits must reject accidental huge uploads.

## Function/RPC Contract

### create_doctor_invite_code

Actor: doctor.

Input:

- Optional display label or note for the doctor only.

Behavior:

- Verifies actor role is doctor.
- Creates a short unique code.
- Sets `max_uses = 1`, `used_count = 0`, `status = active`.
- Sets `expires_at = created_at + 7 days`.
- Records audit event.

Output:

- Invite code id.
- Plain code value for the doctor to give to the patient.
- Expiration timestamp.

### revoke_doctor_invite_code

Actor: doctor.

Input:

- Invite code id.

Behavior:

- Verifies the code belongs to the doctor.
- Allows revocation only when active and unused.
- Sets status to `revoked`.
- Records audit event.

### redeem_doctor_invite_code

Actor: patient.

Input:

- Code text.

Behavior:

- Verifies actor role is patient.
- Rejects invalid, expired, revoked, or already used codes.
- Creates `doctor_patient_access` row with status `active`.
- Marks invite as `used` and increments `used_count`.
- Records audit event.
- Must be atomic to prevent double use.

Output:

- Linked doctor display info.
- Access id.

### export_patient_data

Actor: doctor.

Input:

- Patient id.
- Range type: `day` or `month`.
- Selected date or month.
- Mode: `all_data`, `all_data_with_images`, or
  `images_only_with_labels`.

Behavior:

- Verifies doctor role and active doctor-patient access.
- Calculates date range according to export rules.
- Selects only linked patient data in range.
- Sorts entries by timestamp.
- Includes fields according to mode.
- Creates signed image references only when mode requires images.
- Records export request and audit event.
- Does not include base64 images.

Output:

- JSON package matching `export-json.schema.json`.

## Audit Contract

Audit events required for:

- Consent accepted.
- Doctor invite code created.
- Doctor invite code revoked.
- Invite code redeemed.
- Access denied for patient data.
- Doctor export created.
- Photo upload completed or rejected due to size/type.
- Offline pending entry synced.

Audit metadata must not contain raw audio or base64 image data.
